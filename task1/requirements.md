# 1. Функциональные требования

| ID | Требование | Источник / обоснование |
| :--- | :--- | :--- |
| FR‑1 | Платформа должна поддерживать оформление заказа через все каналы (сайт, мобильное приложение, маркетплейсы) с единой логикой ценообразования, промо и доставки, но с возможностью канальных вариаций. | Интервью бизнес-спонсора: каналы как вариации общей модели |
| FR‑2 | Система должна обеспечивать расчёт стоимости заказа с учётом промоакций, скидок, партнёрских условий и динамических правил. | Описание системы; интервью спонсора (промо-механики) |
| FR‑3 | Платформа должна поддерживать жизненный цикл заказа от оформления до доставки, включая статусы, переходы, отклонения и ручные корректировки. | Описание системы; интервью поддержки (статусы, отклонения) |
| FR‑4 | Система должна интегрироваться с внешними платёжными провайдерами для инициации оплаты, обработки возвратов и подтверждения платежей. | Описание системы; интервью поддержки (платёжные вопросы) |
| FR‑5 | Платформа должна интегрироваться со службами доставки для резервирования слотов, отслеживания отправлений и обновления статусов. | Описание системы |
| FR‑6 | Система должна предоставлять единую карточку заказа для внутренних пользователей (поддержка, операторы) с полной историей изменений, состояний и причин отклонений. | Интервью поддержки (необходимость прозрачности) |
| FR‑7 | Платформа должна поддерживать ручные корректировки заказов (статус, состав, доставка) с аудитом изменений. | Интервью поддержки (ручные сценарии) |
| FR‑8 | Система должна обеспечивать синхронизацию данных с маркетплейсами по заказам, статусам, возвратам и остаткам. | Описание системы |
| FR‑9 | Платформа должна поддерживать сценарии частичной отмены, split shipment и изменения сроков доставки с понятным объяснением для клиента и оператора. | Интервью поддержки (болезненные сценарии) |
| FR‑10 | Система должна предоставлять аналитические данные по заказам, статусам, промо и каналам для управленческой отчётности. | Интервью спонсора (единая картина данных) |

| NFR‑3 | Платформа должна обеспечивать наблюдаемость состояния заказа для клиента и поддержки: понятные статусы, причины отклонений, история изменений. | Высокий | Поддержка (прозрачность) |
| NFR‑4 | Система должна быть способна к поэтапной трансформации без остановки бизнеса и с возможностью отката изменений. | Критично | Спонсор (нельзя останавливать бизнес) |
| NFR‑5 | Архитектура должна снижать время вывода типовых коммерческих изменений (промо, партнёрские условия) по сравнению с текущим состоянием. | Высокий | Спонсор (скорость изменений) |
| NFR‑6 | Данные по заказам, статусам, финансам и логистике должны быть согласованы между разными контурами (витрина, отчёты, поддержка) с минимальными расхождениями. | Высокий | Спонсор (единая картина данных) |

| NFR‑8 | Платформа должна быть готова к постепенному отказу от Oracle и других зарубежных компонентов (импортозамещение) без потери функциональности. | Средний | Спонсор (ограничение) |
| NFR‑9 | Архитектурные изменения должны приносить промежуточную бизнес-ценность, а не только отложенный эффект. | Высокий | Спонсор (этапность) |
| NFR‑10 | Должна быть обеспечена прослеживаемость заказа на всех этапах: от создания до финального статуса, включая внешние вызовы и ручные вмешательства. | Высокий | Поддержка (история изменений) |


# 3. Ограничения и допущения

### Бизнес-ограничения
* **Непрерывность процессов**: невозможность остановки бизнеса на время миграции.
* **Снижение рисков**: поэтапная трансформация с контролируемым риском и возможностью отката.
* **Финансы**: ограниченный бюджет (конкретные цифры не указаны, требуют уточнения).
* **Стабильность операций**: сохранение текущих операционных процессов в период изменений.

### Технические ограничения
* **Архитектурное наследие**: существующее монолитное ядро на Java с центральной БД Oracle.
* **Стек технологий**: наличие вспомогательных сервисов на .NET и Python.
* **Связность систем**: смешанные интеграции (прямые вызовы, обмен через БД, асинхронные очереди).
* **Инфраструктура**: ограниченная виртуализация, физические серверы в собственном дата-центре.
* **Объём данных**: накопленные десятки терабайт исторических данных без системного архивирования.
* **Вендорская зависимость**: критическая зависимость от Oracle и других зарубежных компонентов (требуется импортозамещение).

### Допущения (гипотезы, требующие подтверждения)
* **Готовность к инвестициям**: бизнес готов к последовательным вложениям, если каждый этап даёт измеримый эффект.
* **Экспертиза команд**: команды поддержки и разработки обладают достаточным знанием текущих процессов для участия в трансформации.
* **Гибкость партнёров**: внешние партнёры (маркетплейсы, платёжные провайдеры, службы доставки) готовы к изменениям в интеграционных протоколах.

### Архитектурно значимые пробелы (требуют уточнения)
* **Надёжность**: конкретные значения SLA/SLO (доступность, время отклика).
* **Катастрофоустойчивость**: RTO/RPO для критических сервисов.
* **Производительность**: показатели p95/p99 латенции и допустимые пороги.
* **Масштабируемость**: пропускная способность (RPS/TPS) в пиковые нагрузки.
* **Отказоустойчивость**: правила деградации при отказах внешних систем.
* **Консистентность**: требования к согласованности данных (strong vs eventual consistency).
* **Рамки проекта**: точный бюджет и сроки трансформации.
* **Бизнес-метрики**: ожидаемый рост объёма заказов и пользователей.
