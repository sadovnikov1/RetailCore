
## 3. План поэтапной трансформации
### 3.1 Общие принципы трансформации

1. **Не останавливать бизнес** — все изменения вносятся без даунтайма, с возможностью отката на каждом этапе.
2. **Инкрементальность** — каждый этап даёт измеримую бизнес-ценность и не требует завершения всех последующих этапов.
3. **Dual-write / Dual-read** — на переходных периодах новые и старые механизмы работают параллельно для обеспечения бесшовного переключения.
4. **Наблюдаемость** — на каждом этапе усиливается мониторинг и алертинг для контроля за изменениями.
5. **Обратная совместимость** — изменения не должны нарушать работу существующих интеграций и клиентов.

---

### 3.2 Этап 1: Событийный слой и аналитика (0–3 месяца)

#### Цель этапа

Создать фундамент для всех последующих изменений — единый поток событий по заказу и выделенное аналитическое хранилище.

#### Что делаем

1. **Внедрение Kafka:**
   - Разворачиваем управляемый Kafka-кластер (Confluent Cloud / AWS MSK / Yandex Managed Kafka).
   - Создаём топики: `order-events`, `payment-events`, `delivery-events`, `campaign-events`.
   - Настраиваем репликацию (factor = 3) и retention (7 дней для order-events, 30 дней для delivery-events).

2. **Публикация событий из монолита:**
   - Дорабатываем `OrderProcessingService` для публикации событий после успешного завершения транзакции.
   - Внедряем паттерн **Transactional Outbox** для гарантированной доставки (событие сохраняется в таблицу `outbox` в той же транзакции, затем отдельный процесс публикует в Kafka).
   - Начинаем с ключевых событий: `OrderCreated`, `OrderStatusChanged`, `PaymentConfirmed`, `DeliveryReserved`.

3. **Создание аналитического хранилища:**
   - Разворачиваем ClickHouse (или управляемый PostgreSQL для начала).
   - Настраиваем потребление из Kafka в ClickHouse через Kafka Connect или самописный consumer.
   - Создаём базовые витрины: `orders_fact`, `order_status_history`, `delivery_fact`.

4. **Добавление read replica Oracle:**
   - Создаём read replica для промежуточного переключения аналитических отчётов (снижение нагрузки на master).

#### Критерии готовности этапа

- [ ] Kafka кластер развёрнут и работает в production-режиме (мониторинг, алертинг).
- [ ] Монолит публикует события для 100% новых заказов (dual-write).
- [ ] ClickHouse наполняется данными из Kafka (lag < 5 секунд в пиковые часы).
- [ ] Хотя бы 3 ключевых отчёта переключены на ClickHouse.
- [ ] Read replica Oracle настроена и используется для части аналитических запросов.

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Потеря событий при сбое Kafka | Использовать acks=all, репликацию, мониторинг ISR |
| Увеличение латенции checkout из-за публикации | Публикация асинхронная (отдельный поток), не блокирует транзакцию |
| Расхождения между Oracle и ClickHouse | Периодические проверки консистентности, алертинг при расхождениях > 1% |
| Сложность настройки Kafka Connect | Использовать managed-решения (Confluent Cloud) |

#### Ожидаемый бизнес-эффект

- Улучшение наблюдаемости заказа (журнал событий).
- Снижение нагрузки на Oracle (аналитика переходит на ClickHouse).
- Основа для последующих выносов.

### 3.3 Этап 2: Замена интеграций через БД (4–6 месяцев)

#### Цель этапа

Ликвидировать интеграции через staging-таблицы, заменив их на событийную модель через Kafka.

#### Что делаем

1. **Доработка монолита:**
   - Добавляем публикацию событий в топик `order-export` в дополнение к записи в `INT_ORDER_EXPORT` (dual-write).
   - Для механизма retry (бывшая таблица `RETRY_DELIVERY_QUEUE`) внедряем Kafka с Dead Letter Queue (DLQ).

2. **Доработка .NET-сервиса:**
   - Добавляем потребление из Kafka параллельно с чтением из staging-таблиц.
   - Реализуем идемпотентную обработку (используем `event_id` для дедупликации).
   - Настраиваем обработку DLQ для сценариев retry.

3. **Переключение и отключение:**
   - После стабилизации Kafka-потока (2–4 недели dual-write) переключаем .NET-сервис полностью на Kafka.
   - Отключаем запись в `INT_ORDER_EXPORT`.
   - Удаляем таблицу (после периода наблюдения).

4. **Внедрение механизма retry через Kafka:**
   - Настраиваем DLQ для `order-export` с exponential backoff (повторы через 1, 5, 15, 60 минут).
   - Настраиваем алертинг при попадании сообщений в DLQ.

#### Критерии готовности этапа

- [ ] .NET-сервис потребляет 100% событий из Kafka (dual-write период завершён).
- [ ] Запись в `INT_ORDER_EXPORT` отключена и таблица удалена.
- [ ] Механизм retry работает через Kafka DLQ (проверено на тестовых данных).
- [ ] Мониторинг lag и DLQ настроен и алерты работают.

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Потеря событий при миграции | Длительный dual-write период (2–4 недели), мониторинг расхождений |
| Дублирование обработки в .NET | Идемпотентность через `event_id`, проверка на дубли в БД .NET-сервиса |
| Сложность отладки при сбоях | Логирование всех этапов обработки, корреляционные идентификаторы |
| .NET-сервис не успевает обрабатывать поток | Увеличить число consumer’ов (партиции), оптимизировать обработку |

#### Ожидаемый бизнес-эффект

- Снижение связанности с Oracle.
- Улучшение наблюдаемости интеграций (через Kafka).
- Снижение операционной нагрузки (меньше ручных разборов).
- Основа для выноса .NET-сервиса в отдельный контур.

### 3.4 Этап 3: Вынос Status Service (6–9 месяцев)

#### Цель этапа

Создать единый источник правды для статусов заказа (read-часть) и улучшить прозрачность для клиентов и поддержки.

#### Что делаем

1. **Создание Status Service:**
   - Разрабатываем новый микросервис на Go/Java/Kotlin.
   - Проектируем модель данных: `order_status` (order_id, status, previous_status, reason, timestamp, source).
   - Создаём REST API для чтения статусов: `GET /orders/{orderId}/status`, `GET /orders/{orderId}/history`.
   - Добавляем поддержку пагинации и фильтрации.

2. **Наполнение данными:**
   - При старте сервис потребляет все события `OrderStatusChanged` из Kafka за последние 30 дней для построения начального состояния.
   - В реальном времени обновляет статусы из Kafka-потока.

3. **Переключение читателей:**
   - Поэтапно переключаем всех потребителей статусов на API Status Service:
     - Сначала поддержка (внутренние инструменты).
     - Затем клиентский интерфейс (личный кабинет).
     - Затем маркетплейсы (через API Gateway).
     - Затем аналитика (переключение отчётов на новую модель статусов).
   - На время переходного периода сохраняем чтение статусов из Oracle для внутренних write-сценариев.

4. **Унификация модели статусов:**
   - Согласовываем единую модель статусов со всеми бизнес-подразделениями (убираем специальные статусы для отдельных партнёров).
   - Внедряем механизм маппинга статусов для обратной совместимости (если нужно).

#### Критерии готовности этапа

- [ ] Status Service развёрнут и работает в production.
- [ ] Все читатели статусов переключены на API Status Service (кроме критических write-сценариев).
- [ ] Статусы в Status Service синхронизированы с Oracle (расхождения < 0.1%).
- [ ] Модель статусов согласована с бизнесом и задокументирована.

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Рассинхрон между Oracle и Status Service | Периодические проверки консистентности (ежечасный batch-сверка) |
| Сложность миграции исторических данных | Использовать событийный слой (Kafka) для заполнения, не переносить всё сразу |
| Сопротивление команд (привыкли к старым статусам) | Поэтапное переключение, обучение, документация |
| Производительность API при большом количестве запросов | Добавить кэширование (Redis), использовать read replicas |

#### Ожидаемый бизнес-эффект

- Единый источник правды по статусам для всех потребителей.
- Улучшение прозрачности для клиентов и поддержки.
- Снижение ручных операций и эскалаций.
- Основа для выноса write-части статусов на следующем этапе.

### 3.5 Этап 4: Вынос Pricing Service (9–12 месяцев)

#### Цель этапа

Вынести расчёт цены и промо в отдельный сервис для ускорения коммерческих изменений и снижения сложности монолита.

#### Что делаем

1. **Анализ и подготовка:**
   - Полный инвентаризация всех правил ценообразования и промо (код, конфиги, Oracle-процедуры).
   - Создание регрессионного тестового набора (не менее 1000 сценариев).
   - Согласование модели ценообразования с бизнесом.

2. **Создание Pricing Service:**
   - Разрабатываем микросервис (Go/Java) с in-memory кэшем правил (Redis).
   - Реализуем синхронное API: `POST /calculate` — расчёт цены для корзины/заказа.
   - Интегрируем с внешним Promo Service (вызов через HTTP/gRPC).
   - Добавляем кэширование результатов (для повторяющихся запросов).

3. **Перенос правил:**
   - Постепенно переносим правила из монолита в Pricing Service:
     - Сначала базовые правила.
     - Затем региональные исключения.
     - Затем маркетплейс-специфичные правила.
   - Для каждого перенесённого правила — валидация на тестовых данных.

4. **Переключение:**
   - Используем feature flag в монолите для переключения между старым и новым механизмом.
   - Начинаем с небольшого процента трафика (canary release, 1-5-20-50-100%).
   - В случае ошибок — автоматический откат на старую логику.

5. **Удаление legacy:**
   - После стабилизации (не менее 1 месяца 100% трафика) удаляем старую логику из монолита.
   - Отключаем Oracle-процедуры для промо.

#### Критерии готовности этапа

- [ ] Pricing Service развёрнут и работает в production.
- [ ] 100% трафика checkout использует Pricing Service.
- [ ] Регрессионные тесты проходят без ошибок.
- [ ] Oracle-процедуры для промо отключены и удалены.

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Регрессии из-за сложности бизнес-логики | Полное покрытие тестами (unit + integration + regression), A/B-тестирование |
| Увеличение латенции checkout (сетевой вызов) | Кэширование в Redis, in-memory кэш в сервисе, оптимизация сетевых вызовов |
| Ошибки при переносе правил из Oracle-процедур | Поэтапный перенос с валидацией на каждом шаге |
| Недоступность Pricing Service | Fallback на кэшированные данные (stale cache), fallback на монолит (на время переходного периода) |

#### Ожидаемый бизнес-эффект

- Ускорение вывода новых промо-механик (изменение правил в изолированном сервисе).
- Снижение сложности монолита.
- Возможность масштабировать ценообразование независимо.

### 3.6 Этап 5: Дальнейшая декомпозиция (12–18 месяцев)

#### Цель этапа

Вынести оставшиеся критические компоненты и завершить переход к событийно-ориентированной архитектуре.

#### Что делаем

1. **Вынос Payment Orchestration:**
   - Выделение платёжного сервиса, интеграция с внешними провайдерами через Kafka.
   - Внедрение идемпотентности и механизмов компенсации (Saga pattern).

2. **Вынос Delivery Routing:**
   - Выделение сервиса маршрутизации доставки.
   - Интеграция с внешними службами доставки через Kafka.

3. **Вынос Checkout (создание заказа):**
   - Поэтапный вынос бизнес-логики оформления заказа.
   - Использование Saga или Orchestration для координации (создание заказа → оплата → резервирование → доставка).

4. **Отказ от Oracle:**
   - Миграция данных на PostgreSQL / облачные БД (постепенно, по доменам).
   - Перевод всех сервисов на использование новых БД.
   - Полное отключение Oracle.

5. **Внедрение мультирегиональности:**
   - Настройка активной-активной репликации для критических сервисов.
   - Внедрение глобального балансировщика нагрузки.

#### Критерии готовности этапа

- [ ] Все ключевые сервисы вынесены из монолита.
- [ ] Oracle полностью выведена из эксплуатации.
- [ ] Платформа поддерживает мультирегиональное развёртывание.
- [ ] Архитектура готова к масштабированию и новым бизнес-сценариям.

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Сложность распределённых транзакций (Saga) | Использование проверенных паттернов, оркестрация через Temporal/Camunda |
| Потеря данных при миграции с Oracle | Тщательное планирование миграции, dual-write период |
| Сложность отладки распределённой системы | Внедрение распределённой трассировки (Jaeger/Zipkin) |

#### Ожидаемый бизнес-эффект

- Полная готовность к масштабированию (до 50 000+ заказов в день).
- Возможность быстро запускать новые каналы и партнёрские сценарии.
- Снижение TCO за счёт отказа от дорогого Oracle и локального железа.
- Платформа становится основой для новых сервисов и экосистемы.
