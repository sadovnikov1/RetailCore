### 2.2 Границы и зоны риска

В ходе разработки целевой архитектуры были определены зоны ответственности и оценки рисков для каждого компонента системы. Ниже приведено описание того, какие части системы на данном этапе остаются без изменений, какие выносятся первыми, а какие будут трансформироваться позже, с обоснованием каждого решения.

---

#### 2.2.1 Зона 1: Checkout (создание заказа)

**Решение:** остаётся в монолите на всём протяжении первого этапа трансформации.

**Обоснование:**

Оформление заказа — это критический с точки зрения бизнеса write-поток. Любой сбой в этой зоне напрямую отражается на выручке, конверсии и клиентском опыте. Вынос этой логики на раннем этапе сопряжён с неприемлемо высокими рисками, поскольку:

- Требуется абсолютная точность воспроизведения всей бизнес-логики (цена, промо, наличие, маршрутизация) — любые расхождения приведут к финансовым потерям.
- Вокруг checkout выстроены многочисленные зависимости: интеграции с платежами, логистикой, маркетплейсами, внутренними системами.
- Существующая логика содержит множество исторических исключений и ветвлений, которые не задокументированы и не покрыты тестами.
- Требуется сохранение синхронного характера вызовов для обеспечения пользовательского опыта (время ответа критично).

**Что делаем вместо выноса:**

На первом этапе мы не трогаем ядро checkout, но добавляем публикацию событий в Kafka **после** успешного завершения транзакции. Это позволяет начать строить read-модели и событийно-ориентированные интеграции без изменения критической логики.

**Границы:**

- Checkout остаётся единственным источником правды для write-операций на первом этапе.
- Все изменения checkout проходят строгий процесс регрессионного тестирования.

---

#### 2.2.2 Зона 2: Payment Orchestration (управление платежами)

**Решение:** остаётся в монолите, но подготавливается к будущему выносу через событийную модель.

**Обоснование:**

Управление платежами — второй по критичности процесс после checkout. Платёжный поток включает:

- Создание платёжной сессии.
- Взаимодействие с внешними платёжными провайдерами.
- Обработку подтверждений и возвратов.
- Обновление статусов заказа в зависимости от оплаты.

Любая ошибка в этом процессе приводит либо к финансовым потерям (повторные списания, потеря денег), либо к клиентскому недовольству (заказ не подтверждён, хотя деньги списались). Вынос этого компонента требует высокой зрелости событийной инфраструктуры и механизмов идемпотентности.

**Что делаем вместо выноса:**

Внедряем публикацию платёжных событий (`PaymentInitiated`, `PaymentConfirmed`, `PaymentFailed`) в Kafka. Это позволяет:

- Создать отдельную read-модель для финансовой аналитики.
- Передать управление статусами оплаты в Status Service (когда он появится).
- Начать накапливать данные для будущего выноса платёжной оркестрации.

**Границы:**

- Монолит остаётся единственным исполнителем платёжных операций.
- Платёжные события публикуются после завершения транзакции (dual-write).
- Финансовая консистентность обеспечивается на уровне Oracle-транзакций.

---

#### 2.2.3 Зона 3: Статусы заказа (выносятся первыми, но только read-часть)

**Решение:** выносится в отдельный Status Service, но на первом этапе только read-часть.

**Обоснование:**

Управление статусами — идеальный кандидат для первого выноса, потому что:

- Это **read-heavy** нагрузка. Статусы читаются часто (клиент, поддержка, маркетплейсы, аналитика), но пишутся относительно редко.
- Вынос чтения не затрагивает критический write-поток.
- Проблемы с непрозрачностью статусов — одна из главных болей поддержки и бизнеса.
- Статусная модель может быть выделена как отдельный домен без сильных связей с остальной логикой checkout.

**Как реализуем:**

1. Создаём новый сервис `Status Service` с собственной базой данных (PostgreSQL).
2. Монолит продолжает **писать** статусы в Oracle (как и раньше), но **дополнительно** публикует событие `OrderStatusChanged` в Kafka.
3. Status Service потребляет события из Kafka и обновляет свою базу данных (read-модель).
4. Все системы, которым нужно **читать** статусы (поддержка, клиентский интерфейс, маркетплейсы, аналитика), переключаются на API Status Service.
5. Монолит пока продолжает читать статусы из Oracle для внутренних нужд (поскольку write-логика ещё завязана на Oracle).

**Что остаётся в монолите:**

- Write-логика статусов (установка, обновление) — остаётся в монолите до второго этапа.
- Чтение статусов для внутренних write-сценариев (например, при подтверждении заказа).

**Риски:**

- Рассинхрон между Oracle и Status Service на переходном этапе (eventual consistency). Митигация: использовать механизмы проверки консистентности и алертинга при расхождениях.
- Сложность миграции исторических данных по статусам. Митигация: потребление всех событий из Kafka за последние N дней при старте сервиса.

---

#### 2.2.4 Зона 4: Интеграции через базу данных (.NET-сервис)

**Решение:** заменяются на событийную модель через Kafka.

**Обоснование:**

Интеграция через staging-таблицы в Oracle — одна из самых болезненных точек текущей архитектуры. Она создаёт:

- Сильную связанность между монолитом и .NET-сервисом.
- Неявные зависимости и сложность отладки.
- Нарушение принципов изоляции данных.
- Трудности с масштабированием .NET-сервиса (он привязан к Oracle).

Этот участок подходит для замены, потому что:

- Это изолированный интеграционный слой, не влияющий на критический checkout.
- Можно внедрять параллельно с существующим механизмом (dual-write).
- .NET-сервис уже имеет опыт работы с асинхронными механизмами (очереди, batch).

**Как реализуем:**

1. Добавляем в монолит публикацию событий в топик `order-export` в дополнение к записи в `INT_ORDER_EXPORT` (dual-write).
2. .NET-сервис дорабатывается для потребления из Kafka (параллельно с чтением из таблиц).
3. После стабилизации Kafka-потока переключаем .NET-сервис полностью на Kafka.
4. Отключаем запись в `INT_ORDER_EXPORT` и удаляем таблицу.
5. Для механизма retry используем Kafka с Dead Letter Queue (DLQ) вместо таблицы `RETRY_DELIVERY_QUEUE`.

**Границы:**

- .NET-сервис полностью переходит на Kafka в течение этапа 2.
- Все интеграции через БД ликвидируются.

**Риски:**

- Потеря событий при миграции. Митигация: длительный период dual-write (не менее 2 недель).
- Необходимость сохранения обратной совместимости на время перехода. Митигация: .NET-сервис должен уметь читать и из таблиц, и из Kafka.

---

#### 2.2.5 Зона 5: Аналитика и отчётность

**Решение:** выносится на ClickHouse / отдельное хранилище, потребляющее события из Kafka.

**Обоснование:**

Аналитика — это read-only нагрузка, которая не влияет на write-поток. Вынос аналитики даёт несколько преимуществ:

- Снижение нагрузки на операционную Oracle.
- Улучшение производительности отчётов.
- Создание единой согласованной модели данных для бизнеса.
- Возможность строить отчёты в реальном времени (streaming).

**Как реализуем:**

1. Создаём ClickHouse кластер (или управляемый PostgreSQL) как целевое хранилище для аналитики.
2. Настраиваем потребление из Kafka (топики `order-events`, `payment-events`, `delivery-events`) в ClickHouse через Kafka Connect или самописный consumer.
3. Строим витрины данных и материализованные представления для типовых отчётов.
4. Поэтапно переключаем отчёты на новое хранилище:
   - Сначала новые отчёты.
   - Затем наиболее критичные существующие отчёты.
   - В конце — все оставшиеся.
5. Отключаем прямые SQL-запросы к операционной Oracle.

**Границы:**

- ClickHouse становится единственным источником для аналитических отчётов.
- Oracle больше не используется для аналитических запросов (кроме критических, требующих строгой консистентности — переводятся в последнюю очередь).

**Риски:**

- Расхождения между операционной БД и ClickHouse на переходном этапе. Митигация: использовать проверки консистентности и алертинг.
- Дополнительные затраты на инфраструктуру ClickHouse.

---

#### 2.2.6 Зона 6: Promo и ценообразование

**Решение:** выносятся во второй очереди (этап 4), после стабилизации событийного слоя и read-моделей.

**Обоснование:**

Ценообразование и промо — одна из самых сложных и ветвящихся частей монолита. Она включает:

- Несколько механизмов расчёта (старый, под маркетплейсы, региональный).
- Fallback на Oracle-процедуры.
- Сложные бизнес-правила с множеством исключений.
- Интеграцию с внешним Promo Service.

Вынос этой зоны требует:

- Полного понимания всех бизнес-правил и исключений.
- Наличия событийного слоя для синхронизации данных.
- Высокой зрелости тестирования (регрессионные тесты, A/B-тестирование).
- Готовности к синхронным вызовам через сеть (дополнительная латенция).

**Что делаем до выноса:**

- Внедряем публикацию событий `CampaignUpdated` при изменении промо-правил.
- Строим read-модель для цен в Redis (для быстрого доступа).
- Постепенно переносим правила из Oracle-процедур в код Pricing Service.

**Границы:**

- Pricing Service выносится только после полной стабилизации событийной инфраструктуры.
- На время выноса сохраняется возможность отката на старую логику (feature flag).

**Риски:**

- Высокий риск регрессий из-за сложности бизнес-логики. Митигация: полное покрытие тестами, A/B-тестирование, канареечный релиз.
- Увеличение латенции checkout из-за сетевых вызовов. Митигация: кэширование (Redis), асинхронное обновление цен.
- Необходимость миграции правил из Oracle-процедур. Митигация: поэтапный перенос с сохранением обратной совместимости.

|------|------------------------------|---------------|--------------------------|
| Checkout | Остаётся в монолите | Высокий риск регрессий при изменениях | Строгий регрессионный тестинг, feature flags |
