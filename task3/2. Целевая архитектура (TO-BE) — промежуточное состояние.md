# 2. Целевая архитектура (TO-BE) — промежуточное состояние

Целевая архитектура представляет собой реалистичное промежуточное состояние на горизонте 12-18 месяцев. Она не является полной переписью системы, а описывает эволюционные изменения с чётким разделением:

- **Что остаётся в монолите** — критический write-поток (создание заказа, оплата, резервирование) до полной стабилизации событийного слоя.
- **Что изолируется** — статусы, интеграции через БД, аналитика.
- **Что выносится первыми** — событийный слой, read-модели, аналитическое хранилище.

## 2.1 Описание целевой архитектуры

### Слой 1: Каналы и внешние интерфейсы

- Интернет-магазин, мобильные приложения, маркетплейсы — без изменений.
- API Gateway добавляется для маршрутизации и аутентификации (опционально на первом этапе).

### Слой 2: Монолитное ядро (сохраняется с изменениями)

**Остаётся:** создание заказа, расчёт цены (временно), Payment Orchestration, Delivery Routing.

**Изменяется:**
- Добавляется публикация событий в Kafka при каждом значимом изменении.
- Убирается запись в staging-таблицы (INT_ORDER_EXPORT) — заменяется на Kafka.
- Убираются прямые вызовы Oracle-процедур для промо — заменяются на вызов Pricing Service (на позднем этапе).

### Слой 3: Событийная шина (Kafka) — НОВЫЙ

- **Топик order-events** — все изменения заказа.
- **Топик delivery-events** — статусы доставки от внешних партнёров.
- **Топик payment-events** — статусы платежей.
- **Топик campaign-events** — изменения промо (для обновления кэша).

### Слой 4: Новые сервисы (вынесенные)

- **Status Service** — единый источник статусов, read API.
- **Analytics Storage (ClickHouse/PostgreSQL)** — Data Warehouse для отчётов.
- **Notification Service** — уведомления клиентов (может быть вынесен позже).

### Слой 5: Интеграции (изменённые)

- **.NET Integration Service** — потребляет из Kafka вместо чтения из таблиц.
- **Python Analytics** — потребляет из Kafka в ClickHouse вместо прямых SQL.
- **Promo Service** — остаётся внешним, но вызывается через Pricing Service (в будущем).

### Слой 6: Oracle (сокращается)

**Остаётся для OLTP** (заказы, финансы) на переходный период.

**Убираются:**
- staging-таблицы (перенесены в Kafka).
- тяжёлые аналитические запросы (перенесены в ClickHouse).
- хранимые процедуры для промо (переносятся в Pricing Service).
