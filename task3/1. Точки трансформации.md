# 1. Точки трансформации
На основе анализа текущего состояния выделены 5 точек трансформации, которые могут быть реализованы поэтапно с приемлемым уровнем риска. Каждая точка описывает: проблему, решение, зависимости, риски и предполагаемый порядок реализации.

# Точка 1. Выделение событийного слоя заказа (Order Event Stream)

| Аспект | Описание |
|--------|----------|
| **Проблема** | Статусы и изменения заказа разбросаны по нескольким системам (монолит, .NET, Oracle, batch). Нет единого потока событий для аналитики, поддержки и интеграций. Рассинхрон статусов приводит к непрозрачности для клиента и ручным эскалациям. |
| **Решение** | Внедрить Kafka как единую шину событий заказа. Монолит при каждом значимом изменении (создание, оплата, смена статуса, отмена, изменение доставки) публикует событие в топик `order-events`. <br><br> **Потребители:** <br> • Аналитика (ClickHouse/Data Warehouse) <br> • Поддержка (обновление карточки заказа) <br> • .NET-интеграции (замена staging-таблиц) <br> • Логистика (обновление статусов) |
| **Почему этот участок подходит для изменений** | Это **read-side** изменения — публикация события не изменяет критическую логику checkout. Можно внедрять постепенно: начать с публикации событий в дополнение к существующим механизмам (dual-write), затем переключать потребителей. |
| **Зависимости** | • Требуется инфраструктура Kafka (managed service в облаке или self-hosted) <br> • Нужно доработать монолит для публикации событий (добавить продюсеры в ключевых точках) <br> • Потребители (.NET, Python) должны быть адаптированы |
| **Риски** | • **Риск потери событий при сбое Kafka** — использовать `acks=all` и репликацию <br> • **Риск дублирования** — внедрить idempotent producers и уникальные `event_id` <br> • **Риск увеличения латенции checkout** — публикация должна быть асинхронной (не блокировать транзакцию) |
| **Ожидаемый эффект** | • Единый источник правды по изменениям заказа <br> • Улучшение наблюдаемости <br> • Снижение ручных операций <br> • Основа для выноса аналитики и интеграций |
| **Порядок реализации** | **1-й этап (0–3 месяца)** |

# Точка 2. Выделение аналитического хранилища (read replica + Data Warehouse)

| Аспект | Описание |
|--------|----------|
| **Проблема** | Аналитика напрямую зависит от операционной Oracle. Тяжёлые отчёты конкурируют с OLTP-нагрузкой, создавая блокировки и деградацию производительности. Расхождения в данных между витринами из-за отсутствия единого источника. |
| **Решение** | Создать read replica Oracle для промежуточного этапа. Параллельно начать строить Data Warehouse на ClickHouse или PostgreSQL, наполняя его из Kafka (order-events). Переключить аналитические отчёты на новое хранилище. |
| **Почему этот участок подходит для изменений** | Это read-only нагрузка, не влияет на критический write-поток. Можно делать поэтапно: сначала добавить read replica (быстрое решение), затем мигрировать отчёты на новое хранилище. |
| **Зависимости** | • Требуется инфраструктура для ClickHouse/PostgreSQL <br> • Нужно реализовать ETL/streaming из Kafka в хранилище <br> • Команда аналитики должна адаптировать запросы |
| **Риски** | • Расхождения между операционной БД и новым хранилищем на переходном этапе (eventual consistency) <br> • Дополнительные затраты на инфраструктуру |
| **Ожидаемый эффект** | • Снижение нагрузки на операционную Oracle <br> • Улучшение производительности отчётов <br> • Единая согласованная модель данных для бизнеса |
| **Порядок реализации** | **1–2-й этапы (3–6 месяцев)** |


# Точка 3. Замена интеграций через БД на событийную модель

| Аспект | Описание |
|--------|----------|
| **Проблема** | .NET-сервис интегрируется с монолитом через staging-таблицы в Oracle (INT_ORDER_EXPORT, RETRY_DELIVERY_QUEUE). Это создаёт сильную связанность, неявные зависимости и трудности с отладкой. |
| **Решение** | Заменить обмен через таблицы на Kafka. Монолит публикует события в топик order-export. .NET-сервис потребляет события и обрабатывает их. Для механизмов retry использовать Kafka с Dead Letter Queue (DLQ) вместо таблицы RETRY_DELIVERY_QUEUE. |
| **Почему этот участок подходит для изменений** | Это изолированный интеграционный слой. Можно внедрять параллельно с существующим механизмом (dual-write в таблицу и Kafka), затем переключить .NET-сервис на Kafka и отключить запись в таблицу. |
| **Зависимости** | • Доработка монолита для публикации событий вместо записи в таблицу <br> • Доработка .NET-сервиса для потребления из Kafka <br> • Настройка DLQ и мониторинга |
| **Риски** | • Потеря событий при миграции — использовать dual-write период <br> • Необходимость сохранять обратную совместимость на время перехода |
| **Ожидаемый эффект** | • Снижение связанности с Oracle <br> • Улучшение наблюдаемости интеграций <br> • Возможность выносить .NET-сервис независимо |
| **Порядок реализации** | **2-й этап (4–6 месяцев)** |


# Точка 4. Выделение сервиса статусов (Status Service)

| Аспект | Описание |
|--------|----------|
| **Проблема** | Логика статусов размазана по монолиту, .NET-сервису, batch-процедурам и ручным вмешательствам. Нет единого источника правды по жизненному циклу заказа. Специальные статусы для отдельных партнёров. |
| **Решение** | Выделить отдельный сервис Status Service, который: <br> (1) хранит единую модель статусов и переходов <br> (2) принимает события изменения статуса от всех систем (через Kafka) <br> (3) предоставляет API для чтения статусов <br> (4) содержит бизнес-правила переходов (какие переходы допустимы). |
| **Почему этот участок подходит для изменений** | Статусы — это read-heavy нагрузка, её можно выносить без изменения write-логики. Начать можно с чтения: все системы читают статусы через API Status Service, а пишут пока по-старому (dual-write). Затем переключить запись. |
| **Зависимости** | • Требуется миграция исторических данных по статусам <br> • Нужно согласовать модель статусов со всеми потребителями <br> • Доработка монолита для использования API |
| **Риски** | • Риск рассинхрона на переходном этапе — использовать dual-write с проверкой консистентности <br> • Сложность согласования модели статусов со всеми партнёрами |
| **Ожидаемый эффект** | • Единый источник правды по статусам <br> • Упрощение поддержки <br> • Снижение ручных вмешательств <br> • Основа для прозрачности клиенту |
| **Порядок реализации** | **3-й этап (6–9 месяцев)** |


# Точка 5. Выделение логики ценообразования и промо (Pricing Service)

| Аспект | Описание |
|--------|----------|
| **Проблема** | Расчёт цены и промо — одна из самых сложных и ветвящихся частей монолита. Есть несколько механизмов (старый, под маркетплейсы, региональный). Fallback на Oracle-процедуры. Изменения требуют много времени и несут высокий риск. |
| **Решение** | Выделить Pricing Service как отдельный микросервис. Он должен: <br> (1) рассчитывать цену с учётом всех правил <br> (2) интегрироваться с внешним Promo Service <br> (3) предоставлять API для монолита (синхронно в checkout) и для других сервисов. <br><br> Начать с выноса read-части (расчёт цены при оформлении), оставив write-часть (применение скидок) пока в монолите. |
| **Почему этот участок подходит для изменений** | Это бизнес-логика, которую можно изолировать. Вынос позволит ускорить изменения промо-механик, не затрагивая весь монолит. Но это high-risk зона, поэтому выносить только после стабилизации событийного слоя. |
| **Зависимости** | • Требуется перенос всех правил и коэффициентов из кода, конфигов и Oracle-процедур <br> • Нужно обеспечить высокую производительность (< 50 мс) для синхронных вызовов |
| **Риски** | • Высокий риск регрессий из-за сложности бизнес-логики <br> • Необходимость полного покрытия тестами <br> • Риск увеличения латенции checkout из-за сетевых вызовов |
| **Ожидаемый эффект** | • Ускорение вывода новых промо-механик <br> • Снижение сложности монолита <br> • Возможность масштабировать ценообразование независимо |
| **Порядок реализации** | **4-й этап (9–12 месяцев)** |
