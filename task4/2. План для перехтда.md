# 2. Cutover Plan для ключевого перехода

В качестве ключевого перехода выбран **переключение .NET-сервиса с интеграции через БД на Kafka** (в рамках Этапа 2). Это один из наиболее критичных и рискованных переходов, так как затрагивает внешних партнёров и логистические процессы.

---

#### 2.1 Общая информация о переходе

| Параметр | Значение |
|----------|----------|
| **Наименование перехода** | Миграция .NET-сервиса с staging-таблиц на Kafka |
| **Цель** | Заменить интеграцию через Oracle-таблицы (`INT_ORDER_EXPORT`, `RETRY_DELIVERY_QUEUE`) на событийную модель через Kafka |
| **Время проведения** | Суббота, 02:00 – 08:00 MSK (6-часовое окно) |
| **Тип перехода** | Переключение с dual-write на exclusive Kafka |
| **Ожидаемый эффект** | Снижение связанности с Oracle, улучшение наблюдаемости интеграций |

---

#### 2.2 Условия запуска перехода (Entry Criteria)

Перед началом переключения должны быть выполнены следующие условия:

- [ ] **Dual-write период завершён:** .NET-сервис работал в режиме dual-write (чтение из таблиц + потребление из Kafka) не менее 2 недель без критических инцидентов.
- [ ] **Мониторинг настроен:** Настроены дашборды для отслеживания lag в Kafka, количества сообщений в DLQ, расхождений между таблицей и Kafka.
- [ ] **Регрессионные тесты пройдены:** Все автоматические тесты для логистических интеграций успешно пройдены на stage-окружении.
- [ ] **Команды готовы:** DevOps, разработчики .NET и Java, а также представитель поддержки находятся на связи.
- [ ] **Роллбэк-план готов:** Подготовлен и проверен план отката на предыдущее состояние.
- [ ] **Дашборды и алерты настроены:** В Grafana добавлены дашборды для мониторинга потоков данных, настроены алерты на аномалии.

| Шаг | Время | Действие | Ответственный | Ожидаемый результат |

#### 2.5 Критерии успеха (Success Criteria)

После завершения переключения и периода наблюдения должны быть выполнены следующие условия:

- [ ] .NET-сервис обрабатывает 100% заказов через Kafka.
- [ ] В `INT_ORDER_EXPORT` не появляются новые записи (таблица статична).
- [ ] Расхождение между количеством заказов в Oracle и количеством обработанных событий в .NET < 0.1%.
- [ ] Время обработки события .NET-сервисом не увеличилось более чем на 10% по сравнению с периодом dual-write.
- [ ] Нет аномального роста обращений в поддержку по логистическим вопросам.
- [ ] Все интеграции с внешними партнёрами работают штатно.

---

#### 2.6 План отката (Rollback)

Если в ходе переключения или в течение 2 часов после него возникают критические проблемы (более 5% ошибок, значительное увеличение lag, рост обращений в поддержку), инициируется откат.

| Шаг | Время | Действие | Ответственный |
|-----|-------|----------|---------------|
| R.1 | T+0 | Принять решение об откате (руководитель миграции) | Руководитель |
| R.2 | T+5 | Остановить .NET-сервис | .NET |
| R.3 | T+10 | Вернуть конфигурацию .NET-сервиса на чтение из таблиц (отключить Kafka) | .NET |
| R.4 | T+15 | Запустить .NET-сервис в старом режиме | .NET |
| R.5 | T+20 | Включить запись в `INT_ORDER_EXPORT` в монолите (feature flag) | Java |
| R.6 | T+30 | Проверить, что новые заказы снова попадают в `INT_ORDER_EXPORT` | Java |
| R.7 | T+45 | Проверить обработку .NET-сервисом заказов из таблиц | .NET |
| R.8 | T+60 | Проверить бизнес-процессы (логистика, статусы, уведомления) | .NET / Support |
| R.9 | T+90 | Провести пост-мортем с командой, зафиксировать причины отката | Руководитель |

**Условия для отката:**

- Более 5% ошибок в .NET-сервисе при обработке событий.
- Заметное увеличение времени обработки (более чем на 50%).
- Потеря данных (события не доставляются, но не попадают в DLQ).
- Рост обращений в поддержку по логистическим вопросам более чем на 30%.

**После отката:**

- Проводится анализ причин сбоя.
- Принимается решение о повторной попытке (исправление дефектов, повторный dual-write).
- Вносится корректировка в план переключения.

#### 2.7 Мониторинг и наблюдаемость

**Дашборды (Grafana):**

- **Kafka dashboard:** lag, throughput, error rate, ISR status.
- **.NET dashboard:** request rate, error rate, processing time (p50, p95, p99).
- **Integration dashboard:** количество успешных/неуспешных вызовов к внешним партнёрам.
- **Data consistency dashboard:** сверка количества заказов в Oracle vs Kafka vs .NET.

**Алерты:**

- Lag > 1000 сообщений в течение 5 минут → предупреждение.
- Lag > 10000 сообщений → критический алерт.
- Ошибки в .NET-сервисе > 5% за 5 минут → критический алерт.
- Расхождение данных > 1% → критический алерт.

**Логирование:**

- Корреляционные идентификаторы для отслеживания цепочки: `order_id` → `event_id` → `request_id`.
- Структурированные логи (JSON) для всех этапов обработки.
