### 1. Roadmap миграции

#### 1.1 Общая структура roadmap

Roadmap представляет собой последовательность этапов трансформации от текущего состояния (AS-IS) к целевому промежуточному состоянию (TO-BE). Каждый этап содержит:

- **Наименование и сроки** — период реализации этапа.
- **Цель** — что должно быть достигнуто по итогам этапа.
- **Ключевые активности** — конкретные шаги и задачи.
- **Зависимости** — от каких этапов/работ зависит выполнение.
- **Критерии завершения** — измеримые условия перехода к следующему этапу.
- **Риски** — потенциальные проблемы и способы их митигации.
- **Бизнес-ценность** — какой эффект получает бизнес после завершения этапа.

---

#### 1.2 Этап 0: Подготовка и анализ (Month 0)

**Цель:** Создать базу для всех последующих изменений, устранить пробелы в понимании системы.

**Ключевые активности:**

1. **Документирование AS-IS:**
   - Завершить архитектурную диагностику (задание 2).
   - Задокументировать все интеграции, зависимости, скрытые сценарии.
   - Создать карту потока данных (data flow map).

2. **Создание регрессионного тестового набора:**
   - Определить 100+ ключевых сценариев оформления заказа.
   - Внедрить автоматические тесты для критических сценариев.
   - Настроить сравнительное тестирование (old vs new) для этапов миграции.

3. **Согласование модели данных:**
   - Унифицировать модель статусов заказа.
   - Согласовать каноническую модель заказа для событийного слоя.
   - Определить владельцев данных (data owners) для каждого домена.

4. **Инфраструктурная подготовка:**
   - Развернуть Kafka (dev/stage/prod).
   - Развернуть ClickHouse / аналитическое хранилище.
   - Настроить мониторинг и алертинг (Prometheus + Grafana).

**Зависимости:** Отсутствуют (начальный этап).

**Критерии завершения:**

- [ ] Документированы все ключевые интеграции и зависимости.
- [ ] Регрессионный тестовый набор покрывает 80% критических сценариев.
- [ ] Модель статусов согласована со всеми бизнес-подразделениями.
- [ ] Kafka и ClickHouse развёрнуты в stage-окружении.

**Риски:**

| Риск | Митигация |
|------|-----------|
| Неполное понимание системы | Привлечение ключевых разработчиков для интервью |
| Отсутствие тестов для регрессий | Постепенное накопление тестов, приоритезация по частоте изменений |
| Задержки с инфраструктурой | Использовать managed-сервисы (Confluent Cloud, Yandex Managed Kafka) |

**Бизнес-ценность:** Снижение рисков на последующих этапах за счёт лучшего понимания системы.

---

#### 1.3 Этап 1: Внедрение событийного слоя и аналитики (Months 1-3)

**Цель:** Создать фундамент для всех последующих изменений — единый поток событий по заказу и выделенное аналитическое хранилище.

**Ключевые активности:**

1. **Внедрение Transactional Outbox в монолите:**
   - Создать таблицу `outbox` в Oracle.
   - Доработать `OrderProcessingService` для сохранения событий в outbox в рамках транзакции.
   - Разработать отдельный процесс (poller), который публикует события из outbox в Kafka.

2. **Настройка Kafka:**
   - Создать топики: `order-events`, `payment-events`, `delivery-events`, `campaign-events`.
   - Настроить retention, репликацию, мониторинг lag.

3. **Настройка ClickHouse:**
   - Создать таблицы для хранения событий заказа.
   - Настроить consumption из Kafka (Kafka Connect / самописный consumer).
   - Создать материализованные представления для базовых отчётов.

4. **Создание read replica Oracle:**
   - Настроить репликацию master → slave.
   - Переключить часть аналитических отчётов на read replica.

**Зависимости:** Этап 0 (подготовка).

**Критерии завершения:**

- [ ] Outbox работает в production, события публикуются для 100% новых заказов.
- [ ] Kafka lag < 5 секунд в пиковые часы.
- [ ] ClickHouse содержит данные по всем заказам за последние 30 дней.
- [ ] 3+ ключевых отчёта переключены на ClickHouse.
- [ ] Read replica Oracle используется для 50% аналитических запросов.

**Риски:**

| Риск | Митигация |
|------|-----------|
| Потеря событий при сбое Outbox | Механизм retry, мониторинг ошибок, алертинг |
| Увеличение латенции checkout | Outbox сохраняется в той же транзакции, публикация асинхронная |
| Расхождения между Oracle и ClickHouse | Ежечасная сверка, алертинг при расхождениях > 1% |

**Бизнес-ценность:**
- Улучшение наблюдаемости заказа.
- Снижение нагрузки на Oracle (аналитика переходит на ClickHouse).
- Основа для последующих выносов.

---

#### 1.4 Этап 2: Замена интеграций через БД (Months 4-6)

**Цель:** Ликвидировать интеграции через staging-таблицы, заменив их на событийную модель через Kafka.

**Ключевые активности:**

1. **Доработка монолита:**
   - Добавить публикацию событий в топик `order-export` в дополнение к записи в `INT_ORDER_EXPORT` (dual-write).
   - Внедрить механизм retry через Kafka DLQ вместо таблицы `RETRY_DELIVERY_QUEUE`.

2. **Доработка .NET-сервиса:**
   - Добавить потребление из Kafka параллельно с чтением из staging-таблиц.
   - Реализовать идемпотентную обработку (дедупликация по `event_id`).
   - Настроить обработку DLQ для сценариев retry.

3. **Переключение и отключение:**
   - Держать dual-write 2–4 недели для наблюдения.
   - Переключить .NET-сервис полностью на Kafka.
   - Отключить запись в `INT_ORDER_EXPORT`.
   - Удалить таблицу после периода наблюдения.

**Зависимости:** Этап 1 (событийный слой).

**Критерии завершения:**

- [ ] .NET-сервис потребляет 100% событий из Kafka.
- [ ] Запись в `INT_ORDER_EXPORT` отключена и таблица удалена.
- [ ] Механизм retry работает через Kafka DLQ.
- [ ] Мониторинг lag и DLQ настроен и алерты работают.

**Риски:**

| Риск | Митигация |
|------|-----------|
| Потеря событий при миграции | Длительный dual-write период (2–4 недели) |
| Дублирование обработки в .NET | Идемпотентность через `event_id` |
| .NET-сервис не успевает обрабатывать поток | Увеличить число consumer’ов, оптимизировать обработку |

**Бизнес-ценность:**
- Снижение связанности с Oracle.
- Улучшение наблюдаемости интеграций.
- Снижение операционной нагрузки.

---

#### 1.5 Этап 3: Вынос Status Service (Months 6-9)

**Цель:** Создать единый источник правды для статусов заказа (read-часть) и улучшить прозрачность для клиентов и поддержки.

**Ключевые активности:**

1. **Создание Status Service:**
   - Разработать микросервис с REST API для чтения статусов.
   - Создать модель данных: `order_status` (order_id, status, previous_status, reason, timestamp, source).
   - Реализовать потребление из Kafka для обновления статусов.

2. **Наполнение данными:**
   - При старте потребить все события `OrderStatusChanged` за последние 30 дней.
   - В реальном времени обновлять статусы из Kafka.

3. **Переключение читателей:**
   - Поэтапно переключить всех потребителей статусов на API Status Service:
     - Поддержка (внутренние инструменты).
     - Клиентский интерфейс.
     - Маркетплейсы.
     - Аналитика.

4. **Унификация модели статусов:**
   - Согласовать единую модель со всеми бизнес-подразделениями.
   - Внедрить механизм маппинга для обратной совместимости.

**Зависимости:** Этап 1 (событийный слой).

**Критерии завершения:**

- [ ] Status Service развёрнут в production.
- [ ] Все читатели статусов переключены на API Status Service.
- [ ] Расхождения статусов с Oracle < 0.1%.
- [ ] Модель статусов согласована и задокументирована.

**Риски:**

| Риск | Митигация |
|------|-----------|
| Рассинхрон между Oracle и Status Service | Ежечасная сверка, алертинг при расхождениях |
| Сложность миграции исторических данных | Использовать Kafka для заполнения |
| Производительность API | Кэширование (Redis), read replicas |

**Бизнес-ценность:**
- Единый источник правды по статусам.
- Улучшение прозрачности для клиентов и поддержки.
- Снижение ручных операций.

---

#### 1.6 Этап 4: Вынос Pricing Service (Months 9-12)

**Цель:** Вынести расчёт цены и промо в отдельный сервис для ускорения коммерческих изменений.

**Ключевые активности:**

1. **Анализ и подготовка:**
   - Инвентаризация всех правил ценообразования и промо.
   - Создание регрессионного тестового набора (1000+ сценариев).
   - Согласование модели ценообразования с бизнесом.

2. **Создание Pricing Service:**
   - Разработать микросервис с in-memory кэшем правил (Redis).
   - Реализовать синхронное API для расчёта цены.
   - Интегрировать с внешним Promo Service.

3. **Перенос правил:**
   - Постепенно перенести правила из монолита в Pricing Service.
   - Для каждого правила — валидация на тестовых данных.

4. **Переключение:**
   - Использовать feature flag для переключения трафика (canary: 1% → 5% → 20% → 50% → 100%).
   - Автоматический откат при ошибках.

5. **Удаление legacy:**
   - Удалить старую логику из монолита.
   - Отключить Oracle-процедуры для промо.

**Зависимости:** Этап 1 (событийный слой), Этап 3 (статусы).

**Критерии завершения:**

- [ ] Pricing Service развёрнут в production.
- [ ] 100% трафика checkout использует Pricing Service.
- [ ] Регрессионные тесты проходят без ошибок.
- [ ] Oracle-процедуры для промо отключены.

**Риски:**

| Риск | Митигация |
|------|-----------|
| Регрессии из-за сложности логики | Полное тестирование, A/B-тестирование, канареечный релиз |
| Увеличение латенции checkout | Кэширование, оптимизация сетевых вызовов |
| Ошибки при переносе правил из Oracle | Поэтапный перенос с валидацией |

**Бизнес-ценность:**
- Ускорение вывода новых промо-механик.
- Снижение сложности монолита.
- Возможность масштабировать ценообразование независимо.

---

#### 1.7 Этап 5: Дальнейшая декомпозиция (Months 12-18)

**Цель:** Вынести оставшиеся критические компоненты и завершить переход к событийно-ориентированной архитектуре.

**Ключевые активности:**

1. **Вынос Payment Orchestration:**
   - Выделение платёжного сервиса.
   - Внедрение идемпотентности и Saga pattern.

2. **Вынос Delivery Routing:**
   - Выделение сервиса маршрутизации доставки.
   - Интеграция с внешними службами доставки через Kafka.

3. **Вынос Checkout:**
   - Поэтапный вынос бизнес-логики оформления заказа.
   - Использование Saga или Orchestration.

4. **Отказ от Oracle:**
   - Миграция данных на PostgreSQL / облачные БД.
   - Полное отключение Oracle.

5. **Мультирегиональность:**
   - Настройка активной-активной репликации.
   - Внедрение глобального балансировщика.

**Зависимости:** Все предыдущие этапы.

**Критерии завершения:**

- [ ] Все ключевые сервисы вынесены из монолита.
- [ ] Oracle полностью выведена из эксплуатации.
- [ ] Платформа поддерживает мультирегиональное развёртывание.

**Риски:**

| Риск | Митигация |
|------|-----------|
| Сложность распределённых транзакций | Использовать Temporal/Camunda для оркестрации |
| Потеря данных при миграции с Oracle | Тщательное планирование, dual-write период |
| Сложность отладки | Распределённая трассировка (Jaeger) |

**Бизнес-ценность:**
- Полная готовность к масштабированию.
- Снижение TCO за счёт отказа от Oracle и локального железа.
- Платформа становится основой для новых сервисов.
